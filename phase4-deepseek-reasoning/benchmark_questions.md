# 第四阶段 Benchmark 题库

> 用同一批题反复测试不同模型、量化方式、上下文长度、推理框架和 reasoning budget。不要频繁换题，否则很难判断优化是否真的有效。

## 使用方式

第一轮只抽 10 题做 smoke test；正式评测建议固定 30-40 题。每道题记录：

| 字段 | 说明 |
|------|------|
| `model` | 模型名与大小 |
| `runtime` | llama.cpp / Ollama / LM Studio / vLLM / SGLang |
| `quant` | FP16 / INT8 / Q4 / AWQ / GPTQ / GGUF 等 |
| `prompt_tokens` | 输入 token 数 |
| `output_tokens` | 输出 token 数 |
| `ttft` | 首 token 延迟 |
| `tokens_per_sec` | decode 速度 |
| `vram` | 峰值显存 |
| `correct` | `1 / 0 / partial` |
| `failure_type` | 算错、代码错、理解错、格式错、超时、OOM、输出过长 |

评分时只看最终答案和必要解释，不要求模型暴露完整思维链。

## A. 数学推理题

### M01. 小数比较

**题目**：`9.11` 和 `9.8` 哪个更大？请简短说明。

**标准答案**：`9.8` 更大，因为 `9.8 = 9.80`，而 `9.80 > 9.11`。

### M02. 鸡兔同笼

**题目**：鸡和兔一共有 35 只，脚一共有 94 只。鸡和兔各有多少只？

**标准答案**：鸡 23 只，兔 12 只。

### M03. 两次折扣

**题目**：一件商品先涨价 20%，再降价 20%。最终价格相对原价变化多少？

**标准答案**：变为原价的 `96%`，相对原价下降 `4%`。

### M04. 工程问题

**题目**：甲单独完成一项工作要 6 天，乙单独完成要 9 天。两人一起做 2 天后，剩下的由甲单独完成，还需要几天？

**标准答案**：还需要 `8/3` 天，约 `2.67` 天。

### M05. 概率题

**题目**：一个袋子里有 3 个红球和 2 个蓝球，不放回抽 2 个球。抽到两个红球的概率是多少？

**标准答案**：`3/10`。

### M06. 数列

**题目**：数列 `2, 6, 12, 20, 30, ?` 的下一项是多少？说明规律。

**标准答案**：`42`，第 n 项可看作 `n(n+1)`。

### M07. 方程

**题目**：解方程：`3(x - 2) + 4 = 2x + 9`。

**标准答案**：`x = 11`。

### M08. 反直觉平均数

**题目**：一辆车前半程速度 40 km/h，后半程速度 60 km/h。全程平均速度是多少？

**标准答案**：`48 km/h`，不是 50。

### M09. 取模

**题目**：`2^100 mod 7` 等于多少？

**标准答案**：`2`。

### M10. 组合

**题目**：从 8 个人中选 3 人组成小组，其中 Alice 和 Bob 不能同时被选中。有多少种选法？

**标准答案**：`50`。

## B. 代码推理题

### C01. Python 闭包

**题目**：下面代码输出什么？为什么？

```python
funcs = []
for i in range(3):
    funcs.append(lambda: i)

print([f() for f in funcs])
```

**标准答案**：`[2, 2, 2]`。闭包捕获的是变量 `i`，循环结束后 `i` 为 2。

### C02. 默认参数

**题目**：下面代码输出什么？

```python
def add_item(x, items=[]):
    items.append(x)
    return items

print(add_item(1))
print(add_item(2))
print(add_item(3, []))
```

**标准答案**：依次输出 `[1]`、`[1, 2]`、`[3]`。

### C03. 时间复杂度

**题目**：判断下面函数的时间复杂度，并说明原因。

```python
def f(n):
    i = 1
    total = 0
    while i < n:
        for j in range(n):
            total += i + j
        i *= 2
    return total
```

**标准答案**：`O(n log n)`。

### C04. 二分边界

**题目**：写一个函数 `lower_bound(nums, target)`，返回有序数组中第一个 `>= target` 的下标。如果不存在，返回 `len(nums)`。

**评分点**：循环不死、边界正确、空数组正确。

### C05. LRU 设计

**题目**：设计一个固定容量的 LRU Cache，说明 `get` 和 `put` 如何做到 `O(1)`。

**评分点**：哈希表 + 双向链表；访问后移动到最近端；容量满时淘汰最久未使用节点。

### C06. Bug 定位

**题目**：下面函数想判断括号是否匹配，但有 bug。指出问题并修复。

```python
def valid(s):
    stack = []
    pairs = {')': '(', ']': '[', '}': '{'}
    for ch in s:
        if ch in pairs.values():
            stack.append(ch)
        elif ch in pairs:
            if stack[-1] != pairs[ch]:
                return False
            stack.pop()
    return len(stack) == 0
```

**标准答案**：遇到右括号时需要先判断 `stack` 是否为空，否则 `stack[-1]` 会报错。

### C07. SQL 推理

**题目**：表 `orders(user_id, amount, created_at)`，写 SQL 查询 2025 年每个用户的订单总额，只保留总额大于 1000 的用户。

**评分点**：`WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01'`，`GROUP BY user_id`，`HAVING SUM(amount) > 1000`。

### C08. 并发问题

**题目**：两个线程同时执行 `counter += 1`，为什么最后结果可能小于执行次数？如何修复？

**评分点**：读-改-写不是原子操作；使用锁、原子变量或单线程聚合。

### C09. 测试用例设计

**题目**：为 `parse_int(s)` 设计测试用例。函数要求解析带可选正负号的十进制整数，非法输入抛异常。

**评分点**：正常数、负数、`+` 号、空字符串、只有符号、前后空格、非数字、溢出边界。

### C10. 代码生成

**题目**：用 Python 写一个函数 `top_k_frequent(words, k)`，返回出现频率最高的 k 个单词；频率相同时按字典序升序。

**评分点**：统计频率；排序 key 为 `(-freq, word)`；返回前 k 个。

## C. 逻辑与多步推理题

### L01. 排名

**题目**：A、B、C 三人赛跑。已知 A 不是第一，B 不是第二，C 不是第三，且没有并列。如果 B 不是第一，三人的名次是什么？

**标准答案**：C 第一，A 第二，B 第三。

### L02. 真假话

**题目**：三个人 A、B、C 中只有一个说真话。A 说：“B 说假话。” B 说：“C 说假话。” C 说：“A 和 B 都说假话。” 谁说真话？

**标准答案**：B 说真话。

### L03. 家庭关系

**题目**：小李说：“小王的父亲是我父亲的儿子。”如果小李没有兄弟，小王和小李是什么关系？

**标准答案**：小王是小李的孩子。

### L04. 日历

**题目**：如果某年的 3 月 1 日是星期一，那么同年的 3 月 31 日是星期几？

**标准答案**：星期三。

### L05. 路径规划

**题目**：从 A 到 B 必须经过 C 或 D。经过 C 要 5 分钟，经过 D 要 7 分钟。C 路线有 30% 概率堵车并额外花 10 分钟，D 路线稳定。按期望时间选哪条？

**标准答案**：C 路线期望 `8` 分钟，D 路线 `7` 分钟，应选 D。

### L06. 条件推理

**题目**：如果系统开启缓存，则重复请求会更快。现在重复请求没有更快。能否推出系统没有开启缓存？

**标准答案**：不能严格推出。可能缓存未命中、缓存失效、瓶颈不在缓存等。这是逆否命题误用。

### L07. 约束满足

**题目**：三个任务 X、Y、Z 要排成一列。X 必须在 Y 前面，Z 不能排第一。所有可行顺序有哪些？

**标准答案**：`XZY`、`XYZ`。

### L08. 信息不足

**题目**：一个班平均分 80 分，去掉最高分后平均分下降到 78 分。能否确定最高分是多少？

**标准答案**：不能，缺少班级人数。

### L09. 反事实

**题目**：如果所有 A 都是 B，所有 B 都是 C。某个对象不是 C。能推出什么？

**标准答案**：它不是 B，也不是 A。

### L10. 优先级

**题目**：服务出现高延迟，同时 GPU 利用率很低、CPU 很高、请求队列很长。优先排查什么？

**标准答案**：优先排查 CPU 侧瓶颈、调度/序列化/tokenization、请求队列和数据传输，而不是先认为模型计算太慢。

## D. 长上下文与服务压测题

### S01. 长文复用

**题目**：给模型输入同一篇 3000-5000 字文档，连续问 3 个不同问题。比较第一次和后两次的 TTFT。观察 prefix/context cache 是否生效。

**评分点**：记录 prompt_cache_hit 或框架日志；比较首 token 延迟。

### S02. Prefill-heavy

**题目**：输入一段很长文档，只要求输出 3 条要点。记录 prefill 时间和 decode 时间。

**评分点**：能区分输入长导致的 prefill 成本和输出短导致的 decode 成本。

### S03. Decode-heavy

**题目**：输入一句话，让模型写一篇 800 字说明文。记录 tokens/s 和输出 token 数。

**评分点**：decode 阶段占主要成本。

### S04. 并发压测

**题目**：用同一组 8 个短问题，分别以并发 1、2、4、8 请求服务，记录 TTFT、TPOT、总吞吐。

**评分点**：观察 continuous batching 是否提高吞吐，以及延迟是否上升。

### S05. 量化质量

**题目**：同一个模型的 FP16 与 INT4 版本分别回答 M01-M10。比较正确率、输出长度、速度、显存。

**评分点**：不能只看速度，要同时记录质量变化。

### S06. Reasoning Budget

**题目**：同一批题分别限制最大输出 512、2048、8192 token。比较正确率和平均成本。

**评分点**：找出哪些题需要更长 thinking，哪些题不值得。

### S07. 多采样

**题目**：对 M02、M04、M08、C04、L02 各采样 5 次，用多数投票或人工 verifier 选最终答案。

**评分点**：记录正确率提升是否值得 5 倍生成成本。

### S08. 框架对比

**题目**：同一模型、同一量化格式，在两个本地框架中跑同一批题。比较 TTFT、tokens/s、显存、失败率。

**评分点**：优先保持变量一致；如果格式不同，要在备注中写清。

### S09. OOM 边界

**题目**：逐步增加上下文长度，找到当前模型和量化配置下的最大可用上下文。

**评分点**：记录失败点、显存峰值、是否能通过 KV cache 量化或降低 batch 解决。

### S10. 输出格式稳定性

**题目**：要求模型对 M01-M05 只输出 JSON：`{"answer": "...", "brief_reason": "..."}`。测试普通采样和低温采样下格式是否稳定。

**评分点**：格式正确率、答案正确率都要记录。

## 最小正式评测组合

时间不够时，使用这 15 题：

```text
M01, M02, M03, M04, M08,
C01, C02, C04, C06, C10,
L01, L02, L06,
S04, S06
```

这组题能同时覆盖：简单陷阱、多步计算、代码边界、逻辑推理、并发服务、reasoning budget。
