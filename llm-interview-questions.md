## 一、大模型应用开发一面（基础 + 工程）

### 1. 自我介绍

**一句话答案**：按「我是谁 → 做过什么 → 最擅长什么 → 为什么匹配这个岗位」四段式，1–2 分钟讲完。

**展开**：
- 突出与岗位最相关的项目：做过哪些 LLM 应用、是否负责过训练/推理优化、上线指标（QPS、首 token 延迟、幻觉率下降多少）。
- 避免流水账，每段只留最亮的 1–2 个数字。
- // 结尾留一个钩子，比如「最近在折腾 MLA 的 KV 压缩」，方便面试官顺着提问。

---

### 2. 了解 DeepSeek-R1 吗？介绍一下。

**一句话答案**：R1 是面向**推理**的模型，靠「少量冷启动 SFT + 大规模 RL」让模型学会长链路思考、自我验证与纠错。

**展开**：
- **训练范式**：冷启动 SFT（稳定输出格式） → RL（GRPO 等）激发推理 → 再 SFT + RL 做一轮对齐。
- **核心卖点**：在数学、代码、逻辑推理基准上对标 o1 级别，同时开源权重。
- **工程继承**：MLA（低显存注意力）、MoE、FP8 训练、DualPipe 流水等高效组件。

---

### 3. ⭐ R1 的 MLA 是如何实现 KV-Cache 节约的？

**一句话答案**：把每层每头完整的 K/V 压缩成一个**低维 latent**缓存，推理时再按需解压回各头，KV-Cache 显著变小。

**展开**：
- 传统 MHA 缓存维度：`2 × L × H × d_head`（K 和 V 各一份）。
- MLA 缓存维度：`L × d_c`，其中 `d_c << H × d_head`，节约可达 10× 以上。
- **位置编码解耦**：把带 RoPE 的一小部分维度单独留出（`d_rope`），压缩部分不参与旋转，避免压缩破坏相对位置信息。
- 代价：多了一次上投影矩阵乘法，但这部分可以吸收进 QK 计算。

---

### 4. R1 在 SFT 时冷启动的目的？

**一句话答案**：给 RL 一个**稳定的初始策略**，避免 RL 在「格式乱 + 语言差 + 不会推理」的起点上瞎撞。

**展开**：
- 没冷启动直接 RL：能探索出推理路径，但输出经常中英混杂、格式跳跃、反复跳出思维链。
- 用少量（几千到几万条）高质量 long-CoT 样本做 SFT 后，RL 只需优化「推得更准」，不用再兼顾「说得清」。

---

### 5. ⭐ 位置编码是什么？解释 RoPE。

**一句话答案**：Transformer 本身对顺序无感，**位置编码**把「第几个 token」的信息注入进来；**RoPE** 通过对 Q/K 做位置相关的旋转，让 `<q_m, k_n>` 自然携带**相对位置** `m - n`。

**展开**：
- **做法**：把 Q/K 向量两两分组当作复数，每对按角度 `θ_i = 10000^(-2i/d)` × 位置 `m` 旋转。
- **数学性质**：`<R_m q, R_n k> = <q, R_{n-m} k>`，即点积只依赖相对位置。
- **优点**：
  - 相对位置天然支持；
  - 外推较友好（配合 NTK/YaRN 可扩到 100K+）；
  - 与 FlashAttention、KV-Cache 无缝兼容。
- // 常见变体：NTK-aware scaling、YaRN、LongRoPE。

---

### 6. ⭐ 一个 14B 的模型，推理和训练要多少显存？

**一句话答案**：
- **推理**：BF16 ≈ 28 GB，INT8 ≈ 14 GB，INT4 ≈ 7–10 GB，再加 KV-Cache。
- **训练（全参 Adam）**：往往 **200 GB+**，必须多卡 ZeRO/FSDP；LoRA/QLoRA 单卡 24–48 GB 就能跑。

**展开**（每参数占用经验公式）：

| 场景 | 每参数字节数 | 14B 估算 |
|---|---|---|
| FP16/BF16 权重 | 2 B | 28 GB |
| INT8 权重 | 1 B | 14 GB |
| INT4 权重 | 0.5 B | 7 GB |
| 全参 Adam 训练（BF16 权重 + FP32 梯度/一二阶矩） | ~16 B | ~224 GB |
| LoRA（只训 rank=8 矩阵） | ~2 B（冻结）+ 小量可训 | 24–48 GB |

别忘了还要加：**KV-Cache**（正比于 `batch × seq × layers × d_model`）、激活值、框架 overhead。

---

### 7. 显存的占用和哪些因素相关？

**一句话答案**：参数量 × 精度 + 梯度 + 优化器状态 + 激活值 + KV-Cache + 并行策略 + 梯度检查点。

**记忆口诀**：
- **训练大头**：激活值（受 seq_len、batch 影响最大）+ 优化器状态。
- **推理大头**：权重 + KV-Cache（长上下文时 KV-Cache 甚至超过权重）。

---

### 8. ⭐ 大模型灾难性遗忘是什么？如何解决？

**一句话答案**：学了新任务/新领域后，**原有通用能力明显下降**。解法核心是「**小步快跑 + 混训 + 持续监控**」。

**常用缓解手段**：
1. 领域数据 + 通用数据 **混训**（常见比例 1:1 ~ 1:4）；
2. **更小学习率**、**更少 epoch**；
3. 用 **LoRA / QLoRA** 等参数高效微调，只动一小部分参数；
4. **Replay / Rehearsal**：保留一部分预训练/通用样本持续回放；
5. **多任务联合训练**；
6. 每轮结束跑**通用基准**（MMLU、C-Eval、GSM8K 等）监控退化。

---

### 9. ⭐ BF16、FP16、FP32 对比

| 格式 | 总位数 | 指数位 | 尾数位 | 数值范围 | 训练稳定性 | 显存 |
|---|---|---|---|---|---|---|
| FP32 | 32 | 8 | 23 | 很大 | 最稳 | 最大 |
| FP16 | 16 | 5 | 10 | 窄（±65504） | 易上/下溢，需 loss scaling | 一半 |
| BF16 | 16 | **8** | 7 | 与 FP32 同阶 | 稳（无需 loss scaling） | 一半 |

**结论**：现代 LLM 训练 **BF16 是默认**，FP16 主要出现在老硬件或推理上；FP32 用作 master weights 或数值敏感部分。

---

### 10. Adam、AdamW 原理

**一句话答案**：Adam 用一阶矩（动量）+ 二阶矩（方差）自适应每个参数的学习率；**AdamW 把 weight decay 从梯度里解耦出来**，正则更纯粹，训练 Transformer/LLM 更稳。

**展开**：

```
m_t = β1·m_{t-1} + (1-β1)·g_t          # 一阶矩
v_t = β2·v_{t-1} + (1-β2)·g_t²          # 二阶矩
m̂ = m_t / (1-β1^t), v̂ = v_t / (1-β2^t)  # 偏差修正
θ_t = θ_{t-1} - lr · m̂ / (√v̂ + ε)       # Adam

AdamW 额外做：θ_t -= lr · wd · θ_{t-1}  # 解耦的 weight decay
```

Adam 的 L2 正则会被自适应学习率缩放，实际强度随参数漂移；AdamW 修掉了这个坑。

### 11. ⭐ DeepSpeed 的三个阶段（ZeRO）

| 阶段 | 切分内容 | 显存节省 | 通信代价 |
|---|---|---|---|
| ZeRO-1 | 优化器状态 | 中 | 低 |
| ZeRO-2 | 优化器状态 + 梯度 | 高 | 中 |
| ZeRO-3 | 优化器状态 + 梯度 + **参数本身** | 极高 | 高（每层 all-gather） |

**选择建议**：能 ZeRO-2 跑就 ZeRO-2，ZeRO-3 留给真的塞不下的超大模型。再进一步就是 **FSDP**（PyTorch 原生等价物）。

## 二、大模型应用开发二面（Attention & 训练工程）

### 1. ⭐ 写一下 attention 公式（代码实现角度）

**公式**：

\[
\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}} + M\right) V
\]

**实现步骤**（PyTorch 伪代码）：

```python
def attention(x, W_q, W_k, W_v, W_o, mask):
    Q, K, V = x @ W_q, x @ W_k, x @ W_v          # [B, T, d]
    Q, K, V = split_heads(Q, K, V)                # [B, H, T, d_h]
    scores = Q @ K.transpose(-1, -2) / math.sqrt(d_h)
    scores = scores.masked_fill(mask == 0, -1e9)
    weights = scores.softmax(dim=-1)
    out = weights @ V                             # [B, H, T, d_h]
    out = merge_heads(out) @ W_o
    return out
```

推理时配合 **KV-Cache**：当前 step 的 K/V 拼接到历史缓存末尾，避免重复计算。

---

### 2. ⭐ 训练时为什么要 mask，推理时也需要吗？

- **训练**：逐 token 并行，必须用 **causal mask** 防止偷看未来；再叠加 **padding mask** 忽略 `<pad>`。
- **推理**：逐 token 生成，**因果约束天然满足**（后面的 token 还没生成），但实现上通常仍保留 mask 作为安全网（尤其是 prefill 阶段批量处理 prompt 时）。
- **序列长度**：推理 seq_len **不一定等于** 训练 seq_len，受模型上下文窗口、RoPE 外推能力、KV-Cache 显存三者共同限制。

---

### 3. Transformer 和传统 seq2seq 有什么区别？

| 维度 | RNN/LSTM seq2seq | Transformer |
|---|---|---|
| 计算方式 | 顺序 | 并行 |
| 长依赖 | 靠门控，越远越衰减 | 任意两点一步到位 |
| 并行训练 | 差 | 好 |
| 复杂度 | O(n) | O(n²)（对长序列是劣势） |
| 可扩展性 | 有限 | 极好，催生大模型 |

---

### 4. ⭐ 计算 attention 时为什么要除以 √dk？还有哪些 scale？

**核心**：Q、K 各维度独立方差为 1 时，点积方差约为 `d_k`，不缩放会让 softmax 进入**饱和区**（梯度≈0）。除以 √d_k 把方差拉回 1。

**其他常见 scale**：
- **温度缩放**（解码时）：`logits / T`；
- **QK-Norm**：对 Q、K 分别做 LayerNorm/RMSNorm，训练更稳（PaLM、Grok 在用）；
- **Attention logits clipping**：限制 `scores` 范围；
- **Softcap**（Gemma）：`tanh(x/c)·c`；
- **混合精度下高精度累加**：softmax 用 FP32 累加再转回 BF16。

---

### 5. softmax 的作用、公式，为什么减最大值？换别的函数还能这样吗？

**公式**：`softmax(x)_i = exp(x_i) / Σ_j exp(x_j)`

**减最大值**：利用 `softmax(x) = softmax(x - c)`（平移不变性），避免 `exp` 上溢。

**能否换函数**：只有**对常数平移不变**的归一化函数才能这么干。例如 `x / Σx` 就不行（平移会改变结果）。sparsemax、entmax 也有类似技巧但需单独推导。

---

### 6. 数值上溢、下溢

- **上溢**：超过浮点上界 → `inf` / `NaN`；典型场景：`exp(大数)`、softmax 未减最大值、FP16 累加。
- **下溢**：绝对值太小 → 被 flush 成 0 或丢精度；典型场景：梯度太小、`exp(负大数)`、FP16 梯度更新。

**通用解法**：减最大值、log-sum-exp trick、loss scaling（FP16）、BF16、梯度裁剪、稳健初始化。

---

### 7. ⭐ prompt 书写经验；如何判断达标、如何改进？

**好 prompt 五要素**：任务 + 约束 + 输出格式 + 上下文 + 边界条件（+ 少量高质量示例）。

**判断达标要靠数据，不靠感觉**，至少看这几个指标：
- 准确率 / 格式遵循率 / 拒答率 / 幻觉率 / 延迟 / token 成本。
- 固定一组「**回归测试集**」做 A/B。

**改进优先级（从高到低）**：
1. 补充**输出格式**与**约束**（JSON schema、字段清单）；
2. **拆任务**（一步变多步）；
3. 加 **few-shot**；
4. 加**检索上下文**（RAG）；
5. 换模型/调温度。

---

### 8. ⭐ SFT 对哪部分计算 loss？为什么？

**一句话答案**：对话场景下，**只对 assistant 回复计算 loss**，system/user prompt 部分用 `label=-100` mask 掉。

**原因**：
- 目标是让模型学「**怎么回答**」，不是去拟合用户输入；
- 避免在 user 段浪费容量、降低对格式和角色的敏感度；
- 预训练任务才是「无差别 next-token」，SFT 必须更精准。

// 小坑：要把 `<|assistant|>` 这类 role token 前后是否算 loss 设计清楚，否则模型可能学不会主动起手。

### 9. 讲一下 DeepSpeed

**一句话答案**：微软出的**训练 + 推理优化框架**，核心是 ZeRO 显存优化、混合精度、梯度累积、流水并行、Inference Engine。

**主要能力**：
- **ZeRO-1/2/3 / ZeRO-Offload / ZeRO-Infinity**；
- **3D 并行**：DP + TP + PP；
- 混合精度、梯度累积、梯度检查点；
- DeepSpeed-Inference / DeepSpeed-MII（推理加速）；
- 与 HuggingFace Trainer、Accelerate 深度集成。

---

### 10. 训练一个 7B 模型要多少显存？不同 ZeRO 阶段能省多少？

**7B + BF16 + Adam 粗估**：
- 权重 14 GB，梯度 14 GB，一阶矩 14 GB，二阶矩 14 GB，master weights 28 GB → **仅状态就 ~84 GB**，再加激活往往 100 GB+。

**ZeRO 节省（以 N 张卡为例）**：
- ZeRO-1：优化器状态 ÷ N；
- ZeRO-2：再把梯度 ÷ N；
- ZeRO-3：参数也 ÷ N，单卡可做到 ~`84/N` GB 量级。

但实际比例还要看激活、并行策略、通信开销，不能只看参数。

---

### 11. 训练时用了几张卡，遇到什么异常中断？

**面试万能框架**：
- **规模**：N 张卡、什么并行（TP/PP/DP/ZeRO）、跑了多久、吞吐多少 tokens/s。
- **典型故障**：
  - OOM（长样本、KV-Cache、梯度累积导致）；
  - **NCCL 超时**（慢节点、网络抖动）；
  - **loss spike**（脏样本、lr 过大）；
  - 机器掉卡、ECC、磁盘写满、checkpoint 损坏；
  - 混合精度 NaN。
- **应对**：监控（loss/grad_norm/throughput）、断点续训、自动重试、checkpoint 校验、坏样本隔离。

## 三、大模型（LLM）基础面（架构与体系）

### 1. 目前主流的开源模型体系有哪些？

按维度归类回答更有层次：

| 方向 | 代表 |
|---|---|
| 通用基座 | LLaMA 系列、Qwen、Mistral、Gemma |
| 推理增强 | DeepSeek-R1、QwQ、o1 类开源复现 |
| MoE | Mixtral、DeepSeek-V2/V3、Qwen-MoE |
| 多语言/中文强 | Qwen、GLM/ChatGLM、Baichuan、Yi、InternLM、DeepSeek |
| 端侧/轻量 | Phi、Gemma-2B、Qwen-0.5/1.5B、MiniCPM |
| 代码 | CodeLLaMA、DeepSeek-Coder、Qwen-Coder、StarCoder |
| 多模态 | Qwen-VL、InternVL、LLaVA、MiniGPT-4 |

---

### 2. ⭐ prefix Decoder、causal Decoder、Encoder-Decoder 的区别

| 结构 | 可见性 | 代表 | 适合场景 |
|---|---|---|---|
| **Causal Decoder** | 只能看左侧 | GPT、LLaMA、Qwen | 开放式生成、大规模预训练 |
| **Encoder-Decoder** | Encoder 双向，Decoder 自回归 + 交叉注意力 | T5、BART、原版 Transformer | 翻译、摘要等输入→输出映射 |
| **Prefix LM** | 前缀双向可见，生成部分因果 | GLM、UniLM | 兼顾理解与生成 |

---

### 3. 大模型 LLM 的训练目标是什么？

- **主流（Decoder-only）**：**next-token prediction**（自回归语言建模）。
- 其他：
  - BERT：MLM（masked language model）；
  - T5：span corruption（连续片段 mask）；
  - XLNet：排列语言模型；
  - ELECTRA：replaced token detection。
- **为什么 NTP 赢了**：目标统一、数据易组织、天然支持生成、scale 友好。

---

### 4. 涌现能力是什么原因？

**一句话答案**：模型规模、数据规模、算力跨过**某个阈值**后，某些能力**突然**显著出现。

**主流解释（无单一结论）**：
- 参数容量足够后，能把多个子技能**组合**起来完成复杂任务；
- 训练数据覆盖度够，罕见模式也能学到；
- 推理时的 CoT/工具调用提供了「**外挂**」；
- **评测方式**本身有阶跃（如精确匹配 vs 部分分），会放大「涌现」观感（Jason Wei 2022 vs Schaeffer 2023 的争论）。

---

### 5. ⭐ 为什么现在大模型大多是 Decoder-only？

1. **训练目标统一**：NTP 一个 loss 打天下，数据只需拼成一条流；
2. **工程简单**：没有 encoder-decoder 的交叉注意力、mask 设计更干净；
3. **推理部署生态成熟**：KV-Cache、投机解码、vLLM 等都围绕它优化；
4. **Scaling law 验证充分**：GPT-3 / Chinchilla 系列确认 Decoder-only 的扩展性；
5. 理解任务不是强项？→ instruction tuning + RAG + 工具调用**已基本补齐**。

---

### 6. 简单介绍一下大模型（LLMs）

**一句话答案**：在超大规模语料上通过 next-token prediction 预训练的 Transformer 模型，再经 SFT + 偏好对齐 + 工具/检索增强后，具备**问答、写作、代码、抽取、推理、多模态**等通用能力。

---

### 7. 175B、60B、540B 等数字指什么？

**参数量**，B = billion（十亿）。175B ≈ 1750 亿参数。

**但**：参数大 ≠ 一定强，效果同时取决于**数据质量、训练配方、对齐方式、推理成本**（Chinchilla law：参数与数据要匹配）。

---

### 8. LLM 的优点

- 泛化强、零样本/小样本能力好；
- 多任务统一建模（一个模型打多份工）；
- 跨文本/代码/多模态共用底座；
- 大幅降低规则工程与传统特征工程成本；
- 自然语言即 API。

### 9. LLM 的缺点

- 训练/推理**成本高**；
- **幻觉**难完全消除；
- 稳定性、可解释性不足；
- 数据版权、隐私、安全风险；
- 长上下文、多跳检索、强约束执行仍需工程兜底；
- 真实业务效果高度依赖**数据 + 检索 + 工具 + 评测闭环**，不是只靠模型。

---

## 四、大模型（LLM）进阶面（生成策略与长文本）

### 1. 什么是生成式大模型？

能根据上下文**生成连续内容**（文本、代码、图像描述等），本质是学习数据分布并做**条件生成** `p(y | x)`。当前文本领域典型代表是自回归 LLM。

---

### 2. 大模型是怎么让生成的文本丰富而不单调的？

两条腿走路：

- **模型侧**：大规模多样化数据 + 足够容量 + 多样化对齐数据。
- **解码侧**：`temperature` + `top-p/top-k` + `repetition_penalty` + `no-repeat-ngram` + contrastive search。

上线时再加重复惩罚、长度控制、风格约束、后处理。

---

### 3. ⭐ 什么是 LLM 复读机问题？

模型不断重复某些词、短语、句式，甚至陷入死循环，信息增量极低。典型表现：
- 句尾连续复制上一句；
- 列表项 1/2/3 后面开始无限循环；
- 特定句式「首先/其次/总之」反复切换。

### 4. 为什么会出现复读机问题？

1. **训练目标**只关注局部 next-token 概率，没有全局多样性奖励；
2. 训练数据本身有**模板化、重复**模式；
3. 长文本生成**错误累积**（一旦进入重复轨道，越走越窄）；
4. **贪心 / 低温度**解码容易收敛到高概率重复路径；
5. 上下文窗口不足或约束弱。

### 5. ⭐ 如何缓解复读机问题？

**训练侧**：去重、Unlikelihood Training、反重复负样本、DPO/RLHF 把重复作为负偏好。

**解码侧**：
- `repetition_penalty`（一般 1.05–1.2）；
- `no_repeat_ngram_size`（一般 3–4）；
- `top_p=0.9` + `temperature=0.7` 组合；
- Contrastive Search；
- 限制最大生成长度、检测到循环提前停。

**后处理**：规则去重、句段裁剪。

最稳妥：训练 + 解码 + 评测**三位一体**。

---

### 6. Unlikelihood Training

显式给「不希望出现的 token / 重复模式」加**负对数似然惩罚**，而不是只奖励正确 token。

```
L = -log p(y⁺) - α · log(1 - p(y⁻))
```

对减少重复、模板化、违规内容有效，但训练实现比普通 CE 复杂。

### 7. 引入噪声为什么能缓解重复？

打破模型对某条高概率路径的过度确定性，扩大搜索空间。做法：dropout、采样噪声、word dropout、SwitchOut。噪声过大会损伤稳定性和事实性，**权衡**。

### 8. Repetition Penalty

对**已出现过**的 token 再次出现时降低 logit：

```
logit_i /= penalty   (if token_i appeared)
```

简单有效，系数常用 1.05–1.3；过大会让语言变生硬。

### 9. Contrastive Search

每步从 top-k 候选中选：

```
score = (1-α) · p_model(x) - α · max_sim(x, history)
```

**保留高概率 + 惩罚与历史过相似**，比单纯调温度更稳定。

### 10. Beam Search

保留 `B` 个高分候选并行扩展，适合**确定性任务**（机器翻译）。开放式生成容易过度保守、甚至更容易重复，因此**LLM 聊天很少用**。

### 11. Top-K Sampling

每步只在概率最高的 K 个 token 中采样。K 太小保守，太大可能抽到离谱 token。

### 12. ⭐ Nucleus Sampling（Top-P）

保留**累计概率 ≥ p** 的最小 token 集合再采样。比 Top-K 更自适应（尖锐分布时少采，平坦分布时多采）。常用 `p=0.9~0.95`。

### 13. Temperature

缩放 logits：`logits / T`。
- `T < 1`：分布更尖锐、更确定；
- `T > 1`：分布更平滑、更多样；
- `T → 0`：退化为 greedy。

通常和 top-p 一起用，**不要两个都很极端**。

### 14. No Repeat N-Gram Size

禁止生成已出现过的 n-gram，强约束去重。优点直接，缺点过严会误伤合理重复（如人名、代码关键字）。

### 15. 重复率指标怎么做？

- **distinct-n**：生成中 unique n-gram 占比，越高越多样；
- **self-BLEU**：多次生成之间互算 BLEU，越低越多样；
- 重复 n-gram 占比、最长连续重复片段长度；
- 业务自定义模板重复率；
- 人工抽检 + LLM-as-judge 判「信息增量是否不足」。

### 16. 后处理和过滤如何缓解重复？

规则去重、句段裁剪、重复片段删除、格式清洗。**治标不治本**，但上线兜底必备。

### 17. 人工干预和控制

业务规则、模板、约束解码（JSON schema、grammar）、拒答策略、人工审核。客服、金融、医疗等高风险场景，这是**最后一道防线**。

---

### 18. ⭐ LLaMA 输入句子长度理论上可以无限长吗？

**不能**。限制因素：
- 训练时见过的**上下文窗口**（超过后效果迅速衰减）；
- 注意力 **O(n²)** 的计算和显存开销；
- 位置编码的外推能力（即使 RoPE 配合 NTK/YaRN，也有上限）；
- 长距离信息**衰减**与 needle-in-haystack 准确率下降。

「能喂更长」≠「效果仍可靠」。

---

### 19. 什么情况用 BERT，什么情况用 LLaMA/ChatGLM？

| 场景 | 推荐 |
|---|---|
| 分类 / 匹配 / 排序 / NER / 标签明确 / 低延迟 / 低预算 | **BERT 类** |
| 开放式生成 / 多轮对话 / 复杂推理 / 工具调用 / 解释 | **LLaMA / Qwen / GLM** |
| 需要长文本理解 + 结构化抽取 | 大模型 + 约束解码 |

---

### 20. 各个专业领域是否需要各自的大模型？

**不一定**。大部分场景用 **通用模型 + RAG + 工具调用 + 少量领域 SFT** 即可。

**值得训练领域模型的条件**：
- 术语密度高（医疗、法律、金融合规）；
- 流程刚性强、格式必须严格一致；
- 数据不能出域（私有化）；
- 通用模型 + RAG 做完评测后仍显著不够。

---

### 21. ⭐ 如何让大模型处理更长的文本？

四类手段，**通常组合使用**：

1. **扩窗口**：继续训练到更长上下文（32K→128K→1M）；
2. **位置编码外推**：NTK-aware、YaRN、LongRoPE、ALiBi；
3. **高效注意力**：FlashAttention、Ring Attention、稀疏/滑窗、分块注意力；
4. **应用层工程**：切分 + 检索（RAG）+ 摘要 + 多轮规划 + Map-Reduce。

真实业务里，**第 4 点性价比最高**，不是单纯把窗口做大。

---

## 五、大模型（LLM）微调与领域训练面

### 1. ⭐ 在某模型基础上做全参数微调，需要多少显存？

全参 Adam 经验值 **~16 bytes/param**（权重+梯度+一阶+二阶+master）：

| 模型 | 粗估显存（仅状态） | 实际（加激活） |
|---|---|---|
| 7B | ~112 GB | 常需 4–8 卡 |
| 14B | ~224 GB | 常需 8–16 卡 |
| 70B | ~1.1 TB | 多机 + ZeRO-3/FSDP |

资源不够时优先 **LoRA / QLoRA**（7B 常可单卡 24 GB）。

### 2. ⭐ 为什么 SFT 之后感觉 LLM 变傻了？

常见四大原因：

1. **数据太窄**、任务类型单一 → 通用能力被覆盖；
2. **格式太死**（全都逼它回 JSON）→ 自然语言表达退化；
3. **训练轮次过多 / 学习率过大** → 过拟合指令样式；
4. **评估口径变了**：模型更听话了，但创造性下降，你感觉是「变傻」。

**解法**：混训通用数据、降 lr、减 epoch、用 LoRA、同时跑通用基准 + 业务基准。

### 3. SFT 指令微调数据如何构建？

1. **明确任务类型与输出规范**；
2. 收集**高质量**问答 / 流程 / 工具调用 / 多轮对话；
3. 统一**模板**（system/user/assistant、特殊 token、分隔符）；
4. 关键：**去重、纠错、难度分层、边界样本、负例**；
5. 留**验证集**，监控是否过拟合格式。

### 4. 领域 Continue PreTrain 数据怎么选？

**质量 > 数量**。优先：
- 与领域强相关的**行业文档、手册、知识库、规范、案例库、高质 FAQ**；
- 版权、合规清晰；
- 去重、去噪、去模板污染。

避免：重复数据、风格极单一、来源可疑的数据。

### 5. ⭐ 领域训练后通用能力下降怎么办？

**黄金配方**：
1. 领域 + 通用**混训**（1:1 ~ 1:4，按业务调）；
2. 更小 lr、更少 epoch；
3. 参数高效微调（LoRA/QLoRA）；
4. 保留 **replay buffer**（一部分原始预训练样本）；
5. 每轮评估通用基准 + 领域基准。

### 6. 领域 Continue PreTrain 如何让模型学到更多知识？

**重点不是堆量，是提高「有效 token」的知识密度**：
- 清洗去重、消除模板噪声；
- 按主题 / 难度做 **curriculum learning**；
- 覆盖关键术语与定义；
- 加入**结构化资料**（表格、定义、流程图文本化）；
- 持续评测挑出真正带增益的数据源，剔除「训了等于没训」的部分。

### 7. ⭐ SFT 时基座选 Chat 还是 Base？

| 选择 | 适用 |
|---|---|
| **Base** | 想从头按自己的格式做对齐，保留最大可塑性 |
| **Chat** | 只是想在已有对话能力上贴合业务风格，省事 |

面试加分：基础能力越强、已对齐程度越高，**二次训练越容易训坏**原有能力，需要更小 lr 和更少数据。

### 8. 领域模型微调时，指令和数据格式有什么要求？

- 格式**稳定、边界清楚、角色明确**；
- 统一对话模板（ChatML / LLaMA 格式 / Qwen 格式，选一个坚持用）；
- 明确：system/user/assistant 分隔、特殊 token、多轮拼接方式；
- **只对 assistant 回复算 loss**；
- 工具调用结果如何结构化进样本（function call / tool role）。

### 9. 领域评测集怎么构建？

- **严格与训练集隔离**；
- 覆盖：高频场景、长尾场景、边界场景、失败样本；
- 指标：准确率/召回率 + **格式遵循 + 事实性 + 鲁棒性 + 拒答率 + 人工满意度**；
- **保留通用评测**监控退化（MMLU、C-Eval、GSM8K、HumanEval…）。

### 10. 词表扩增有必要吗？

**通常不是第一优先级**。现代 tokenizer 已经能处理绝大多数新词；扩词表带来：
- 新 embedding 初始化问题；
- 与原模型兼容性问题；
- 训练稳定性下降。

**值得扩的前提**：领域里**大量高频**被严重碎片化的专有词，且确实影响效果或推理效率。

### 11. 如何训练自己的大模型？

标准流程：

```
目标/预算 → 数据采集清洗 → tokenizer/架构选型 → 预训练
→ Continue PreTrain → SFT → 偏好对齐(DPO/RLHF) → 评测 → 部署
```

**90% 的团队不需要从零训练**，基于开源基座做增量训练性价比最高。真正难点：
- 数据工程；
- 训练稳定性（loss spike、NaN）；
- 显存与并行策略；
- 评测闭环。

### 12. 训练中文大模型的经验

- 高质量中文语料稀缺，**清洗去重尤其重要**；
- 关注中英文比例、知识类与对话类平衡；
- 分词碎片化问题（某些开源 tokenizer 对中文不友好）；
- 格式污染严重（广告、导航栏、HTML 残留）；
- Benchmark 要用 C-Eval、CMMLU、MMLU-Pro 中文等代表性组合；
- 长句、全角标点、表格、古文等特殊场景需额外评测。

### 13. 指令微调的好处

让基座从「会续写」变成「会按要求做事」：
- 指令遵循；
- 输出格式稳定；
- 多轮交互；
- 业务可用性。

相比从头训练，**性价比高、迭代快**，最适合把模型拉向产品形态。

### 14. ⭐ 预训练 vs 微调，哪个阶段注入知识？

| 阶段 | 注入什么 |
|---|---|
| **预训练** | 广泛、底层、可迁移的语言模式与世界知识（**容量式**注入） |
| **微调（SFT）** | 任务格式、输出偏好、流程、**少量**高价值领域知识（**行为式**注入） |

**要内化大量领域知识 → Continue PreTrain**；要让它按格式/流程做事 → SFT。

### 15. 学行业知识应该预训练还是微调？

- 想让模型**真正吸收**行业术语、背景、表达习惯 → **Continue PreTrain**；
- 想让它按特定**指令 / 模板 / 流程**工作 → **SFT**；
- 常见路线：**领域 Continue PreTrain → 领域 SFT → 偏好对齐**。

### 16. ⭐ 多轮对话任务如何微调？

关键点：
1. 把多轮历史按**统一模板**串起来；
2. **只对当前轮 assistant 回复**算 loss；
3. 样本覆盖：上下文引用、指代消解、拒答、追问、工具返回、上下文截断；
4. 长对话注意 **position id** 与截断策略，避免训练/推理长度分布不一致。

### 17. 能力劣化、灾难性遗忘怎么回事？

本质：**参数在新分布上被过度拉动**，原能力对应的参数配置被破坏。诱因：
- 数据单一；
- 学习率过大；
- 训练过久；
- 样本质量差；
- 任务目标过窄。

管好：**数据配比 + 训练强度 + 持续评测**。

### 18. 微调模型需要多大显存？

看三点：**全参 vs LoRA/QLoRA、模型大小、序列长度 × batch size**。经验值：
- 7B + QLoRA：单卡 24 GB；
- 7B 全参：多卡或 A100 80G × 2+；
- 14B+：量化 + 梯度检查点 + 分布式几乎必选。

### 19. ⭐ LLM 做 SFT 时在学什么？

**不是新增海量世界知识**，而是把已有能力重新组织成：
- **按指令回答**；
- **按格式输出**；
- **按角色交互**；
- **按流程执行**。

学的是**任务映射 + 行为规范 + 风格偏好**，所以数据格式和标签质量影响巨大。

### 20. 预训练 vs SFT

| 维度 | 预训练 | SFT |
|---|---|---|
| 数据 | 海量原始语料 | 人工整理监督样本 |
| 目标 | 语言分布 + 通用知识 | 任务格式 + 行为约束 |
| 核心关注 | 规模、覆盖 | 质量、模板、任务定义 |
| 量级 | T 级 token | 万–百万级样本 |

### 21. 样本规模增大时 OOM 怎么办？

先定位：**batch 大了？seq 长了？数据预处理吃内存？**

常见解法：
- 减小 **micro-batch**；
- **截断序列**或**动态长度打包**；
- 启用**梯度累积**；
- **梯度检查点**（activation checkpointing）；
- **FlashAttention**；
- **ZeRO / FSDP**；
- **LoRA / QLoRA + 混合精度**；
- 优化 **padding**（packing、sort-by-length）。

### 22. SFT 如何对样本进行优化？

**质量 > 数量**：
- 去重（近似去重 + 精确去重）；
- 纠错、统一口径；
- **边界样本**与**失败修复样本**；
- 难度分层、长度分桶；
- 按任务加权采样；
- 剔除无信息模板噪声。

**1 万条高质量 > 10 万条低质量。**

### 23. 模型参数迭代实验怎么做？

**工程师思维**：
- 固定**评测集**与**核心指标**；
- **一次只改一个变量**（lr / batch / 模板 / LoRA rank / 数据配比）；
- 保留 **config + log + checkpoint + 评测结果**；
- 建系统化实验表（Wandb / MLflow）；
- 避免「感觉变好了」但不可复现。

### 24. 微调大模型的建议

- 先建**强基线**，别一上来就全参；
- **小步快跑**，一次只改一个变量；
- 数据干净 > 模板统一 > 评测可靠 > 调超参；
- 业务目标明确时，**LoRA / QLoRA 往往足够**。

### 25. batch size 太小会怎样？

- 梯度**噪声大**、训练不稳定；
- 收敛慢，loss 抖动明显；
- 个别坏样本影响大；
- 虽然有时小 batch 助于泛化，但过小**得不偿失**。

### 26. batch size 太大会怎样？

- 显存压力直接上升；
- 更新过于**平滑**，泛化反而变差；
- 通常需要**重新调大 lr 和 warmup**（Linear Scaling Rule）；
- 不是越大越好，要找**吞吐 × 稳定性 × 泛化**平衡点。

### 27. batch size 如何设置？

1. 根据显存上限确定 **micro-batch**；
2. 用**梯度累积**凑出目标 **global batch**（常见 1M 左右 tokens / step）；
3. 起步沿用成熟开源配方；
4. 看 **loss 曲线 + 吞吐 + 最终评测**调整，而不是只看速度。

### 28. 微调优化器怎么选？

- 全参 / LoRA / QLoRA：**AdamW 仍是默认强基线**；
- 超大规模 / 特殊硬件：可试 **Adafactor**（省显存）、**Lion**（显存低但需精调 lr）、**Sophia**；
- **前提**：有对照实验支撑，否则别瞎换。

### 29. 哪些因素影响内存使用？

- 参数量、精度；
- 优化器状态；
- batch size、**序列长度**（二次项大头）；
- 是否缓存 KV；
- 激活保存策略、梯度检查点；
- 并行方式；
- 框架额外开销（PyTorch、DeepSpeed、Megatron 的 buffer）。

**常被忽略**：长序列 + 大 batch 才是最容易爆显存的组合。

### 30. 领域预训练用什么数据集？

- 与业务强相关、**清洗过**、合规；
- 行业百科、标准规范、技术文档、客服语料、历史案例、高质 FAQ；
- 中文场景还要看：**来源多样性、去重、时效性、术语覆盖度**。

### 31. 微调数据集如何构建？

从真实任务流出发：
1. 定义样本类型；
2. 设计输入输出模板；
3. 收集**正样本 + 反例 + 边界例 + 失败修复样本**；
4. 训练/验证**严格隔离**；
5. 每批数据都要能回答：**「这批数据要改变模型的哪种行为？」**

### 32. ⭐ loss 突刺的原因和解决？

**原因**：
- 脏样本 / 异常长度；
- 学习率过大、warmup 不足；
- 混合精度溢出；
- 梯度爆炸;
- 数据顺序偏移（shuffle 出问题）；
- 分布式通信异常；
- checkpoint 恢复不完整；
- batch 组成突变（碰巧全是某一类）。

**解法**：
- 降低 lr、加长 warmup；
- **梯度裁剪**（clip_grad_norm=1.0 常用）；
- 检查混合精度 + loss scaling；
- 定位坏样本（记录 sample_id → 复现 batch）；
- 监控 **loss + grad_norm + NaN/inf + throughput**；
- 保留可回滚 checkpoint；
- 可疑 batch **复现回放**。

### 33. 什么是 loss 突刺？

原本平稳下降/波动的 loss 在某些 step **突然大幅升高**，常伴随：
- 梯度范数异常；
- NaN/Inf；
- 评测效果骤降。

偶发小波动正常，**持续或大幅** spike 说明训练不稳定。

### 34. 为什么会出现 loss 突刺？

同 32 题原因列表。面试里要能说出至少 **5 条不同层面**的诱因（数据 / 数值 / 学习率 / 通信 / checkpoint）。

### 35. loss 突刺如何解决？

同 32 题解法。工程上额外强调：
- **自动检测**：grad_norm > 阈值就告警/回滚；
- **可回滚**：保留最近 N 个 checkpoint；
- **复现能力**：记录 seed、dataloader 状态、sample_id；
- **坏样本隔离**：过滤掉引发 spike 的 batch 再继续。

| 策略 | 特点 | 典型用途 |
|---|---|---|

## 附录 C：一页纸「背不下来就背这个」

- **RoPE**：Q/K 按位置旋转，点积自然带相对位置。
- **MLA**：K/V 压缩成低维 latent，大幅省 KV-Cache。
- **为啥除 √dk**：不除 softmax 进饱和，梯度消失。
- **softmax 减最大值**：平移不变性，防 exp 上溢。
- **BF16**：指数位=FP32，不用 loss scaling，LLM 训练默认。
- **AdamW**：weight decay 解耦，训练 Transformer 更稳。
- **ZeRO-1/2/3**：分别切 优化器 / +梯度 / +参数。
- **SFT 只对 assistant 算 loss**：学「怎么答」不是「拟合输入」。
- **灾难性遗忘**：混训 + 小 lr + LoRA + 持续评测。
- **复读机**：训练去重 + Unlikelihood + 解码 rep_penalty / top-p / contrastive。
- **长文本**：扩窗口 + RoPE 外推 + 高效注意力 + RAG，**第四条最划算**。
- **loss spike**：lr/warmup/clip/混合精度/脏样本，五线排查。
